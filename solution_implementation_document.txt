AUP Problem 2 - Detailed Implementation Document

Title
Offline, On-Prem Software License Protection: 4-Layer Hybrid Architecture

Purpose
Protect an on-prem software license from tampering, falsification, or bypass, while remaining realistic to deploy in constrained, offline environments.

Scope
This document describes a complete implementation plan that combines:
- Hardware-bound license enforcement
- Trusted execution for secret handling
- Kernel-level control for anti-tamper
- Detection and deception for attacker telemetry

Reference 4-Layer Stack (from solution diagram)
- Layer 0 (TPM Hardware Root of Trust): EK identity, PCR registers, seal/unseal operations
- Layer 1 (SGX Enclave / Trusted Execution): license validation, key handling, core logic execution
- Layer 2 (Kernel Driver / Control Flow): debugger detection, DLL blocking, VM/sandbox detection
- Layer 3 (Application / Detection): honeypot functions, AI anomaly detection, silent kill triggers

Schema Alignment Notes
- License signature can be ECDSA or Ed25519 (both fit the design)
- Add explicit binary hash check for anti-patch
- Keep offline encrypted log buffer for telemetry in Layer 3

Non-Goals
- Total prevention against a nation-state attacker
- Always-on online verification

Requirements (Tools, SDKs, and Dependencies)

Rust Toolchain
- Rust stable (rustup, cargo)
- Recommended: clippy, rustfmt

Rust Crates (core)
- serde, serde_json (license schema)
- ed25519-dalek or p256 (license signatures)
- sha2 (binary hash check)
- aes-gcm (optional payload encryption)
- rand (key material generation)
- thiserror, anyhow (error handling)
- tracing, tracing-subscriber (logging)
- clap (CLI for issuer)

TPM Layer (Layer 0)
- Windows: TPM Base Services (TBS) API or tpm2-tss
- Optional tools: tpm2-tools (for local testing)

Enclave Layer (Layer 1)
- Intel SGX SDK (if real enclave is required)
- Rust SGX bindings or FFI to SGX SDK
- For demo-only: simulated enclave module in Rust

Kernel/Driver Layer (Layer 2)
- Windows: Windows Driver Kit (WDK) if real driver is required
- For demo-only: user-space stubs and checks in Rust

Detection + Telemetry (Layer 3)
- Local encrypted log buffer (implemented in Rust)
- Optional offline anomaly model (simple thresholds in Rust)

Build and Packaging
- CMake or vcpkg only if native libraries are used
- PowerShell scripts for demo automation (optional)


1) Threat Model and Requirements

Threats
- License file modification (users, modules, expiry)
- License replay on another machine
- Binary patching to skip validation
- Reverse engineering to extract secrets
- Debugger or DLL injection attacks
- VM/sandbox usage for snapshots and replay

Constraints
- On-prem deployment with limited or no internet
- Minimal operational overhead for the client
- Must run on commodity hardware where possible

Requirements
- Strong integrity of license data
- Machine binding with integrity checks
- Offline validation capability
- Detect and report tampering attempts


2) License Data Model and Cryptography

2.1 License Payload (JSON structure)
Example fields:
- license_id
- customer_id
- product_id
- modules[]
- max_users
- valid_from
- valid_to
- environment_type
- machine_binding
- issuer
- license_version
- binary_hash

2.2 Integrity
- Use vendor private key to sign the payload
- Use a public key embedded in the application for verification

2.3 Confidentiality (optional)
- Encrypt sensitive fields with AES-256-GCM
- Decryption key is released by TPM only on a trusted machine

2.4 Anti-Rollback
- Include a monotonic counter or last_validated timestamp in a secure log
- Reject older license payloads


3) Layer 0 - TPM Hardware Root of Trust

Goal
Bind the license to the physical machine and boot integrity state.

Core Components (diagram mapping)
- EK identity
- PCR registers
- Seal/unseal operations

Implementation Steps
1) Installation
   - Detect TPM 2.0
   - Generate a random license secret
   - Seal the secret to TPM PCR values (0,1,7)
2) Runtime
   - On app start, request TPM unseal
   - If PCR values match, TPM releases the secret
   - If PCR values changed (tampered boot chain), unseal fails

Key Results
- License secret cannot be used on other machines
- Boot integrity changes break license use


4) Layer 1 - Enclave (Trusted Execution)

Goal
Protect license logic and secrets from OS and debugger access.

Core Components (diagram mapping)
- License validation
- Key handling
- Core logic execution

Implementation Steps
1) Build a trusted enclave
   - Compile core license verification into enclave
   - Sign enclave with vendor key
2) Startup
   - CPU creates enclave and verifies identity
   - If tampered, enclave load fails
3) Runtime
   - Enclave receives encrypted license payload
   - Enclave decrypts using TPM released key
   - Enclave validates signature, expiry, modules, limits
   - Returns only VALID or INVALID to host

Key Results
- No plaintext license secrets outside enclave
- Debugger and memory dumps see only encrypted data


5) Layer 2 - Kernel / Driver Control Layer

Goal
Prevent trivial bypass by patching and debugger attachment.

Core Components (diagram mapping)
- Debugger detection
- DLL blocking
- VM/sandbox detection
- Decoy / real routing (anti-reverse engineering)

Implementation Steps
1) Driver install at system startup
2) Register callbacks
   - Process creation
   - DLL/module load
3) Anti-debug checks
   - Kernel-level debug port checks
   - Kill protected process if debugger detected
4) Module integrity checks
   - Only allow vendor-signed modules
   - Block injection of unsigned DLLs
5) VM detection (optional)
   - Detect known hypervisor signatures
   - Trigger soft response or exit

Key Results
- Bypass attempts are blocked early
- Harder to hook or patch validation logic


6) Layer 3 - Detection and Deception

Goal
Detect misuse, gather intelligence, and waste attacker time.

Core Components (diagram mapping)
- Honeypot functions
- AI anomaly detection
- Silent kill triggers
- Fake endpoints or decoy APIs

Implementation Steps
1) Honeypot license checks
   - Inject decoy functions during build
   - If called, log a tamper event
2) Canary tokens
   - Embed subtle watermarks in license data
   - Detect reuse or unauthorized distribution
3) Anomaly detection
   - Collect usage metrics locally (active users, module access)
   - Lightweight model flags anomalies
4) Event log
   - Encrypted local buffer with tamper detection
   - Send to vendor when connectivity is available

Key Results
- Provides evidence of tampering
- Encourages attacker mistakes


7) Anti-Patch Binary Integrity

Goal
Prevent simple patching to skip checks.

Implementation Steps
1) Include signed hash of vendor binary in license payload
2) On startup, compute SHA-256 of the executable
3) Compare to licensed hash
4) Distribute license checks across multiple code paths

Key Results
- NOP-style bypass attempts fail or trigger alerts


8) Verification Flow (Runtime)

1) Load license file
2) Verify license signature with vendor public key
3) Request TPM to unseal secret
4) Create enclave and verify enclave identity
5) Decrypt payload inside enclave
6) Validate limits, modules, expiry
7) Verify binary hash
8) Run application if valid; otherwise exit gracefully


9) Update, Renewal, and Recovery

- Renewal: issue a new signed license payload
- Migration: vendor re-seals to new TPM if hardware changes
- Revocation: optional offline list, checked on startup


10) Demo / PoC Plan

Minimum viable demo
- License generator
- License validator (TPM mock if unavailable)
- Tamper demonstration
- Patch demonstration (binary hash failure)
- Logging and alert output

Optional extensions
- Simulated enclave interface
- Mock kernel driver callbacks


11) Residual Risks

- Strong attackers with kernel exploits could bypass controls
- No continuous online validation means delayed response
- Some hardware platforms may lack TPM or SGX support


12) Summary

This hybrid architecture protects the license at multiple independent layers:
- TPM binds secrets to the machine
- Enclave keeps validation and secrets private
- Kernel layer blocks patching and debugging
- Detection layer catches and reports misuse

Breaking one layer does not break the whole system. This gives credible, deployable protection for on-prem software licensing in a hackathon-appropriate PoC.
